## 1. 概述

DraMark 是一门专为戏剧、影视及音乐剧的剧本创作设计的纯文本标记语言。其核心目标为：

1. **少输入法切换 (Few IME Switch)**：核心符号 `@`, `%`, `=` 在中文键盘下可直接通过 Shift 键打出，保证编剧创作心流不断。
2. **状态机驱动 (State Machine Driven)**：摒弃繁琐的闭合标签，通过极简的上下文状态推导台词、唱段和舞台动作的作用域。
3. **原生译配支持 (Native Dual-track)**：内建强大的双语译配工作流，一份源文件可编译为演员本、导演本、工作本。
4. **技术与艺术解耦**：将舞台监督的 Cue 标记与角色演出文本彻底分离。

## 2. 与 CommonMark 的关系

DraMark 是 CommonMark 的超集方言。
DraMark 解析器在生成 AST 时，并不破坏 CommonMark 的基础区块（如段落、列表、引用块），而是通过**语义注入**的方式将它们归属于特定角色或动作节点。所有的 CommonMark 格式（如 `*斜体*`、`**加粗**`）在 DraMark 中完全保留。

YAML Frontmatter 在本规范中被定义为**文档配置层（Document Config Layer）**，用于为编辑器/渲染器/编译器提供配置数据；它不属于 DraMark 正文语法本体。

## 3. 核心解析模型：二维状态机

DraMark 的解析引擎基于一个**二维状态机（2D State Machine）**运行。解析器在任何时刻处于以下状态的交叉点：

- **维度一：表演状态 (Performance Context)**
  - `全局动作 (Global Action)`：默认状态。文本被解析为舞台指示或场景描述。
  - `角色状态 (Character)`：文本被解析为当前角色的台词。
- **维度二：音乐状态 (Musical Context)**
  - `对白模式 (Spoken)`：默认状态。正常的戏剧对白。
  - `唱段模式 (Sung)`：进入音乐剧曲目（Musical Number）状态。

状态的切换由核心指令（`@`, `---`, `#`, `$$`）驱动。详见后续章节。

## 4. 文档配置层 (Frontmatter)

文档可由 YAML Frontmatter 块开头，用于提供剧本配置数据。
该配置块在解析前优先提取，并交由上层系统（前端/渲染器/编译器）消费。
解析器应遵循“宽松 + 推荐字段”策略：未知字段允许透传。

```yaml
---
meta:
  title: 悲惨世界 (Les Misérables)
  locale: zh-CN
  version: 0.1.0
casting:
  characters:
    - name: 冉阿让
      actor: 张三
      mic: HM1
      aliases: [24601]
  groups:
    principals:
      members: [冉阿让]
translation:
  enabled: true
  source_lang: en
  target_lang: zh-CN
  render: bilingual # source | target | bilingual
tech:
  mics:
    - id: HM1
      label: Hamlet 主麦
      color: "#4B8BFF"
  sfx:
    - id: SND_THUNDER
      file: thunder.mp3
      desc: 雷声
  lx:
    - id: LX01
      desc: 冷白顶光
      color: "#E6EEFF"
  keywords:
    - token: blackout
      label: 黑场
      color: "#111111"
---
```

### 4.1 推荐配置命名空间

- `meta`：剧本元数据（标题、语言、版本、版权等）
- `casting`：角色清单与分组（角色 ID、展示名、别名、分组映射）
- `translation`：译配开关与渲染策略
- `tech`：技术资源字典（麦克风、音效、灯效、关键词与颜色）

### 4.2 解析器职责边界

- 解析器负责：
  - 提取 frontmatter 原文（供上层做归一化）
  - 提供最小可用元信息（例如 `translation.enabled`）
- 解析器不负责：
  - 严格业务 schema 校验
  - 消费端的呈现/编排策略
- 未知字段必须保留（forward-compatible）

### 4.3 `casting` 关联规则（推荐）

- `group` 只在 `casting.groups` 定义，作为分组单一事实来源（single source of truth）。
- `@角色` 行不携带分组字段；正文仅声明“当前发言者标识”。
- `casting.characters[]` 的最小推荐字段为：
  - `name`（建议唯一；默认检索键）
  - `id?`（仅在重名或跨系统关联时建议提供）
  - `actor?`（演员展示名）
  - `mic?`（角色默认麦克风 ID，建议引用 `tech.mics[].id`）
  - `aliases?`
- `casting.groups.<groupName>.members[]` 可写 `name` 或 `id`；默认按 `name` 匹配，若成员值命中某个 `id` 则按 `id` 优先。
- 当出现重名且未提供 `id` 时，消费端应给 warning，但不阻断解析。
- 兼容优先原则：非重名场景下，不要求提供 `id`；`@角色` 匹配与 group 成员匹配都可直接按 `name` 完成。
- `actor` 与 `mic` 均为可选字段，缺省时不影响解析。

### 4.4 `tech` 最小可执行模型（推荐）

`tech` 推荐包含四类字典：`mics`、`sfx`、`lx`、`keywords`。每类内 `id` 应唯一；未知字段允许透传。

- `tech.mics[]`
  - `id`（必填）
  - `label`（必填）
  - `color?`, `tags?`, `enabled?`
- `tech.sfx[]`
  - `id`（必填）
  - `file`（必填）
  - `desc?`, `color?`, `tags?`, `enabled?`
- `tech.lx[]`
  - `id`（必填）
  - `desc`（必填）
  - `color?`, `tags?`, `enabled?`
- `tech.keywords[]`
  - `token`（必填）
  - `label`（必填）
  - `color?`, `tags?`, `enabled?`

### 4.5 换麦扩展配置（app/扩展层，推荐）

为支持“演员麦克风分配/换麦”而不硬编码单一语言关键词，推荐在 `tech` 下增加可选配置：

- `tech.micDirectives?`
  - 用于声明换麦关键词与同义词（可中英混合、自定义）
  - 示例字段：`switch[]`, `to[]`, `from[]`, `for[]`
- `tech.defaultMicBehavior?`
  - 用于定义默认目标解析行为（如省略目标角色时绑定当前角色）
  - 推荐值：`current-character`

说明：
- 该配置由 app/扩展消费，用于“换麦指令识别层”。
- parser 可继续透传 `frontmatterRaw` 与最小 metadata，不强制内建该语义。

## 5. 角色与台词

### 5.1 角色声明与状态进入

行首的 `@` 后紧跟角色名，解析器即进入该角色的 **Character Context**。

- **语法**：`@<角色名>[<情绪/状态提示>]` 或 `@<角色名>【<情绪/状态提示>】`
- **多角色声明**：允许同一行多个角色标识（例如 `@peter @bobby [aside]`），按实现约定生成主角色与别名列表
- **吞噬规则**：一旦进入角色状态，后续的普通段落、空行、列表（`-`, `1.`）、引用块（`>`）均作为该角色的**台词内容 (Dialogue Content)**，直到遇到状态重置指令。

```markdown
@哈姆雷特
我有两个选择：

- 生
- 还是死
```

### 5.2 动作提示 (Inline Action)

台词内部的身体动作使用 `{}` 包裹。全角 `｛｝` 与半角 `{}` 等价。

- **语法**：`你给我出去！{指着大门}`

## 6. 场景动作与结构分隔

要退出“角色状态”回到“全局状态”（撰写舞台动作），使用以下两种 CommonMark 结构级标记：

1. **Thematic Break (水平线)**：`---`, `***`, `___`
2. **ATX Headings (标题)**：`#`, `##`

水平线在 DraMark 中具有**重置为全局动作**的强语义。

```markdown
@冉阿让
我该何去何从？

---

就在此时，警长沙威猛地推开了大门。 （⚠️ 已回到全局状态，此句为场景动作）

@沙威
你跑不掉了！
```

## 7. 唱段与音乐容器

### 7.1 独立唱段 (Song Block / Musical Number)

由独占一行的 `$$` 开始，独占一行的 `$$` 结束。这标志着解析器进入**唱段模式 (Sung Context)**。
`$$` 是一个**宏观曲目容器**，在其内部，DraMark 的指令依然运作，但作用域被限制在音乐中：

- **遇到 `@角色名`**：不退出唱段，仅切换演唱者（支持多角色对唱/合唱）。
- **遇到 `---`**：不退出唱段，仅将表演状态切回全局动作（表示间奏或音乐伴奏下的舞台调度）。

```markdown
$$
@冉阿让
这颗充满爱的心！
---
% 音乐转为急促的弦乐，起义者们随着鼓点奔跑。

@马吕斯
听吧人民在歌唱！
$$
```

### 7.2 行内唱段 (Inline Song)

在同行内使用 `$<唱词>$`，用于对白与短促唱词的无缝衔接。

## 8. 技术提示标记

技术标记（Tech Cues）用于灯光 (LX)、音响 (SND) 等后台调度，与演员文本严格解耦。

- **行内标记**：`<<LX01 GO>>`。行内标记具有超越 HTML 解析的**词法抢占权 (Lexical Preemption)**，一旦匹配，内部文本作为普通 CommonMark 处理。
- **独立标记**：由 `<<<` 开始， `>>>` 结束的独占行或多行块。一旦匹配，内部文本作为普通 CommonMark 处理。

### 8.1 麦克风分配与换麦扩展（实验性，app/扩展层）

换麦能力当前定义为**应用层扩展**，不是 DraMark 核心语法强制项。推荐支持两种输入形态：

- 基于行内技术提示的换麦指令（例如 `<<MIC SWITCH TO HM2>>`）
- 基于 fenced code block 的技术指令块（例如 <code>```dramark-tech</code>）

推荐默认行为：
- 在角色上下文中，若换麦指令省略目标角色，则默认作用于当前 `@角色`。
- 非角色上下文下省略目标角色，应产出 warning（不阻断正文解析）。
- v1 不定义 `@group` 直接换麦语法；组级批量换麦留待后续以显式 member 映射引入。

兼容性说明：
- 关键词匹配由 `frontmatter.tech.micDirectives` 配置驱动，避免语言锁定。
- 非重名场景默认按角色 `name` 匹配；重名时建议补 `id` 做消歧。

## 9. 注释

注释仅在源文件或特定的工作版渲染目标中可见。

- **行注释**：`% 注释内容`（附着于当前节点，常用于译配直译或注音）。
- **块注释**：`%%` 独占一行开始，`%%` 结束。

## 10. 译配与多语言模式

在 Frontmatter 配置 `translation.enabled: true` 后启用。该模式原生支持原/译文的非对称对齐。

### 10.1 原文与译文的配对 (Translation Pair)

使用等号加空格（`= `）开头的行，表示**原文 (Source Text)**。

**吞噬闭包规则**：
一个 `= ` 声明会持续吞噬后续的普通文本行（包括嵌套的列表、引用块等）作为**译文 (Target Text)**，并将其打包为一组 `TranslationPair`。吞噬会一直持续，直到遇到以下情况之一：

1. 下一个 `= `（下一句原文开始）
2. 下一个 `@`（切换角色）
3. **根级别的** `---` 或 `#`（切回全局动作）
4. `$$`（唱段边界）

```markdown
@冉阿让
= Who am I?
% 直译：我是谁？（注音：hu am ai）
我是谁？
= Can I conceal myself for evermore?
我能否永远把我自己隐藏？
假装我不再是过去的那个我？
```

## 11. 转义字符

支持使用反斜杠 `\` 转义 DraMark 功能字符。
合法的转义序列为：`\@`, `\$`, `\{`, `\}`, `\%`, `\<`, `\=`, `\>`。

## 12. 解析器实现指引与边缘情况裁决 (Edge Cases)

在实现 DraMark 的 Lexer/Parser 时，必须严格遵循以下 6 条宪法级裁决，以消除歧义：

### 裁决一：Frontmatter 配置块的预处理豁免权 (Phase 0 Priority)

虽然 `---` 用于触发状态重置，但**文档首部的 Frontmatter 边界符 `---` 具有第零优先级**。解析器在预处理阶段消耗掉前两个独占行的 `---`，它们**绝对不触发**任何 DraMark 状态重置逻辑。

### 裁决二：状态切换的“容器隔离原则” (Container Isolation)

`---` 和 `#` 具有状态重置和吞噬截断的能力，但**该能力仅在文档根级别（Root-level）有效**。

- 如果 `---` 或 `#` 出现在 CommonMark 的嵌套容器（如 `>` 引用块、`-` 列表项）内部，它们受容器隔离保护，**不触发** DraMark 的状态重置或译配截断。

### 裁决三：唱段容器的隐式穿透 (Song Container Breakout)

结构级标题（`#`, `##`）具备强制穿透 `$$` 容器的最高优先级。如果在 `$$` 内部遇到根级别标题，解析器将隐式闭合当前的 `$$` 唱段，暴露出标题。

### 裁决四：译配目标的数据类型 (Target as Block List)

在 AST 中，`TranslationPair` 的 `target` 属性不是单纯的纯文本字符串，而是 **Array of Blocks（块级节点数组）**。这意味着译文可以完美包含加粗、列表、多段落等格式。

### 裁决五：百分号 `%` 的防误伤词法 (Strict Comment Lexing)

为防止 `%` 误伤日常文本（如“利润下降了 20%”）：

- **行首注释**：若 `%` 是该行的第一个非空字符，整行为注释。
- **行尾/行内注释**：若 `%` 出现文本后，**其左侧必须至少包含一个空白字符**（即 `%`），才能被识别为注释的起点。紧跟在非空白字符后的 `%` 视为普通文本。

### 裁决六：行内技术标记严禁跨行 (No Multi-line Leak)

行内标记 `<< ... >>` 内部**不允许包含换行符（`\n`）**。
_容错机制_：如果当前行扫描到了 `<<` 但行尾前未找到匹配的 `>>`，则放弃解析为技术标记，将其降级为普通文本或移交底层的 HTML 解析器。

## 13. 附录 A：输入法与编辑器体验优化 (IME Tricks)

DraMark 的设计恪守 ASCII 字符底座，但强烈建议各大编辑器（如 VS Code 插件、Obsidian 插件）在应用层实现以下静默映射（IME Tricks），以提供极致的中文创作心流：

当侦测到用户在全角/中文输入法状态下连续键入以下符号时，应**自动替换**为 DraMark 标准标记：

1. **技术标记**：输入 `《《` 自动转为 `<<`，输入 `》》` 自动转为 `>>`。
2. **唱段标记**：输入 `￥￥`（Shift+4 连按）自动转为 `$$`。
3. **动作提示**：输入 `【【` 自动转为 `{`，输入 `】】` 自动转为 `}`。
